Introduction & Environment Setup

What Django is, how MVT works, and getting your first project running.

Day 1 — Introduction & Environment Setup

Today you will learn

  • What Django is and why it's used
  • The MVT (Model-View-Template) architecture
  • Setting up a virtual environment and installing Django
  • Creating your first Django project and running the dev server

Theory

Django is a high-level, free and open-source Python web framework that encourages rapid development and clean, pragmatic design. It follows the philosophy of "batteries included" — an ORM, admin panel, forms, authentication, and security protections all come built-in, so you don't need to hunt for third-party packages to build a real application.

MVT Architecture

Django follows the Model-View-Template pattern (its own take on MVC):

  • Model — defines your data structure and talks to the database (Python classes → database tables)
  • View — contains the business logic; decides what data to fetch and which template to render
  • Template — the HTML file that defines how the data is displayed to the user

Request flow: Browser → urls.py (router) → views.py (logic) → models.py (data) → template (HTML) → back to Browser

Why a virtual environment?

A virtual environment is an isolated Python installation for one project, so its package versions (e.g. Django 4.2) don't clash with another project that might need Django 5.0. It's standard practice for every real Python project, not just Django.

Setup commands terminal
# 1. Check Python is installed
python --version

# 2. Create a virtual environment named 'venv'
python -m venv venv

# 3. Activate it
# Windows:
venv\Scripts\activate
# macOS/Linux:
source venv/bin/activate

# 4. Install Django inside the virtual environment
pip install django

# 5. Confirm the installed version
python -m django --version
Once activated, your terminal prompt shows (venv) — meaning any pip install now happens only inside this project's isolated environment, not system-wide.
Create your first project terminal
# create a new Django project named 'mysite'
django-admin startproject mysite
cd mysite

# run the development server
python manage.py runserver
After running the server, open http://127.0.0.1:8000/ in your browser — you should see Django's default "The install worked successfully!" congratulations page. manage.py is your command center for this project — every Django command runs through it.

Day 3 — URLs & Views (Function-Based)

Today you will learn

  • How Django routes a URL to a view function
  • Writing your first views with HttpResponse
  • Connecting an app's URLs to the project using include()

Theory

A view is a Python function (or class) that takes a web request and returns a web response. The URL dispatcher (urls.py) decides which view handles which URL path. Good practice: each app has its own urls.py, and the project's root urls.py includes it — this keeps routing modular as your project grows.

Basic view blog/views.py
from django.http import HttpResponse

def home(request):
    return HttpResponse("
Welcome to my Blog!
")

def about(request):
    return HttpResponse("This is the about page.")
Every view function takes at least one argument, request — an object holding everything about the incoming request (method, user, GET/POST data). It must always return an HttpResponse (or a subclass of it, like render(), which we'll use from Day 4).
App-level URLs blog/urls.py (new file)
from django.urls import path
from . import views

urlpatterns = [
    path('', views.home, name='home'),
    path('about/', views.about, name='about'),
]
path('', ...) matches the empty path (the app's root). The name='home' argument lets you refer to this URL by name elsewhere (in templates, redirects) instead of hardcoding the path string — so if the URL ever changes, you only update it in one place.
Wiring into the project mysite/urls.py
from django.contrib import admin
from django.urls import path, include

urlpatterns = [
    path('admin/', admin.site.urls),
    path('', include('blog.urls')),   # hand off all other URLs to blog app
]
include() tells Django "for anything matching this prefix, go check that app's own urls.py". Here the prefix is empty, so blog's URLs are mounted at the site root.

Day 4 — Templates & the Django Template Language

Today you will learn

  • Rendering HTML files instead of raw strings using render()
  • Passing data from a view into a template with context
  • Django Template Language: variables, {% if %}, {% for %}

Theory

Instead of writing HTML directly inside views.py (unmaintainable for real pages), Django uses templates — HTML files with special syntax that Django processes before sending them to the browser.

  • {{ variable }} — outputs a value from the context
  • {% tag %} — logic: loops, conditionals, template inheritance
  • context — a Python dictionary passed from the view, containing the data the template needs

By default Django looks for templates inside appname/templates/appname/ — the doubled app name folder avoids naming collisions between apps that both have a file called e.g. detail.html.

Update the view to render a template blog/views.py
from django.shortcuts import render

def home(request):
    posts = [
        {"title": "My First Post", "content": "Hello Django world!"},
        {"title": "Learning Templates", "content": "DTL is powerful."},
    ]
    context = {"posts": posts, "page_title": "Home"}
    return render(request, "blog/home.html", context)
render() combines a template file with a context dictionary and returns a proper HttpResponse — it's the standard way almost every Django view ends.
The template blog/templates/blog/home.html
<!DOCTYPE html>
<html>
<head><title>{{ page_title }}</title></head>
<body>
    <h1>Welcome, {{ page_title }} page</h1>

    {% if posts %}
        <ul>
        {% for post in posts %}
            <li><strong>{{ post.title }}</strong> — {{ post.content }}</li>
        {% endfor %}
        </ul>
    {% else %}
        <p>No posts yet.</p>
    {% endif %}
</body>
</html>
{{ post.title }} works for dictionary keys, object attributes, and even method calls — DTL uses the dot for all of them, unlike Python which needs different syntax for each. Every {% for %} and {% if %} must be explicitly closed with {% endfor %} / {% endif %}.

Day 5 — Template Inheritance & Static Files

Today you will learn

  • Avoiding repeated HTML using {% extends %} and {% block %}
  • Serving CSS, JS, and images with Django's static files system

Theory

Template inheritance lets you define a common layout (navbar, footer, CSS links) once in a base.html and have every other page extend it, overriding only the parts that differ. Static files (CSS, JavaScript, images) are handled separately from templates — Django needs to know where to find them (STATICFILES_DIRS) and what URL prefix to serve them under (STATIC_URL).

Base template blog/templates/blog/base.html
{% load static %}
<!DOCTYPE html>
<html>
<head>
    <title>{% block title %}My Blog{% endblock %}</title>
    <link rel="stylesheet" href="{% static 'blog/css/style.css' %}">
</head>
<body>
    <nav><a href="{% url 'home' %}">Home</a> | <a href="{% url 'about' %}">About</a></nav>

    <main>
        {% block content %}
        {% endblock %}
    </main>

    <footer>© 2026 My Blog</footer>
</body>
</html>
{% load static %} must appear at the top before using {% static %}. {% url 'home' %} generates the correct URL by looking up the name you gave in path(..., name='home') on Day 3 — so links never break even if the actual path string changes.
Child template blog/templates/blog/home.html
{% extends 'blog/base.html' %}

{% block title %}Home — My Blog{% endblock %}

{% block content %}
    <h1>Latest Posts</h1>
    {% for post in posts %}
        <h2>{{ post.title }}</h2>
        <p>{{ post.content }}</p>
    {% endfor %}
{% endblock %}
The child template only defines what's different for this page — everything inside {% block content %} replaces the empty block in base.html; everything else (nav, footer) is inherited automatically.
Static files config mysite/settings.py
STATIC_URL = 'static/'
STATICFILES_DIRS = [BASE_DIR / "static"]   # project-wide static folder (optional)
# App-level static files just need to live in: blog/static/blog/css/style.css
/* blog/static/blog/css/style.css */
body { font-family: Arial, sans-serif; background: #f4f4f4; margin: 0; }
nav { background: #0C4B33; padding: 12px; }
nav a { color: white; margin-right: 15px; text-decoration: none; }

Day 6 — Models & the Django ORM

Today you will learn

  • What the ORM is and why it's useful
  • Defining a model with fields
  • Common field types

Theory

The ORM (Object-Relational Mapper) lets you work with your database using Python classes and objects instead of writing raw SQL. Each model class becomes a database table; each class attribute becomes a column; each instance of the class becomes one row.

Common field types

FieldUse for
CharField(max_length=..)Short text (requires max_length)
TextField()Long text, no length limit
IntegerField()Whole numbers
DateTimeField(auto_now_add=True)Timestamp, set once when created
BooleanField(default=False)True/False flags
EmailField()Email addresses (adds validation)
ForeignKey(Model, on_delete=..)Many-to-one relationship (Day 14)
Defining the Post model blog/models.py
from django.db import models

class Post(models.Model):
    title = models.CharField(max_length=200)
    content = models.TextField()
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)
    is_published = models.BooleanField(default=True)

    def __str__(self):
        return self.title
auto_now_add=True sets the field automatically the first time the object is created and never touches it again. auto_now=True updates the field every time the object is saved. __str__ defines what shows up when you print a Post object or view it in the admin — without it you'd just see Post object (1), which is not useful.

Day 7 — Migrations & Django Shell

Today you will learn

  • How Django turns model changes into database schema changes
  • Using the interactive shell to create and read data

Theory

Migrations are Django's way of version-controlling your database schema. Every time you change a model (add a field, rename one, etc.), you generate a migration file describing that change, then apply it.

  • makemigrations — scans your models for changes and writes a migration file (does NOT touch the database yet)
  • migrate — actually applies migration files to the database
Creating and applying migrations terminal
python manage.py makemigrations
python manage.py migrate
makemigrations output looks like Migrations for 'blog': blog/migrations/0001_initial.py - Create model Post — showing what will change. Always run makemigrations before migrate, and re-run both whenever you edit models.py.
Django shell — creating & reading data terminal
python manage.py shell
from blog.models import Post

# CREATE
p1 = Post(title="My First Post", content="Hello Django!")
p1.save()

# shortcut: create() saves immediately, no separate .save() needed
p2 = Post.objects.create(title="Second Post", content="Learning ORM.")

# READ — get all records
all_posts = Post.objects.all()
print(all_posts)

# READ — get a single record
post = Post.objects.get(id=1)
print(post.title)
Post.objects is the model's manager — the entry point for every database query. .get() expects exactly one matching result and raises an error if zero or multiple rows match — use it only when you're certain of a unique match (like a primary key).

Theory

Django's admin panel is an automatically generated interface for managing your data — one of Django's most famous "batteries included" features. It's meant for site staff/content managers, not end users, but it's also invaluable during development to quickly add test data.

Create a superuser terminal

Customizing the admin display blog/admin.py
from django.contrib import admin
from .models import Post

@admin.register(Post)
class PostAdmin(admin.ModelAdmin):
    list_display = ('title', 'is_published', 'created_at')   # columns shown in the list view
    list_filter = ('is_published', 'created_at')               # sidebar filters
    search_fields = ('title', 'content')                         # adds a search box
    prepopulated_fields = {}    # (used with slug fields, covered later)
@admin.register(Post) is a decorator shortcut equivalent to calling admin.site.register(Post, PostAdmin) afterward. list_display controls which columns show in the table view (by default admin only shows one column, the __str__).

Day 9 — QuerySets Deep Dive (CRUD)

Today you will learn

  • Filtering, ordering, and excluding records
  • Updating and deleting existing records
  • Field lookups (contains, gt, lt, etc.)

Theory

A QuerySet is Django's representation of a database query — it's lazy, meaning the actual SQL only runs when you evaluate it (loop over it, print it, or call list()), not the moment you write the line of code. This lets you chain filters efficiently before hitting the database.

Filtering & ordering shell
from blog.models import Post

# filter — returns a QuerySet (can be empty, one, or many)
published = Post.objects.filter(is_published=True)

# exclude — opposite of filter
drafts = Post.objects.exclude(is_published=True)

# ordering
newest_first = Post.objects.order_by('-created_at')   # '-' means descending
oldest_first = Post.objects.order_by('created_at')

# field lookups — double underscore syntax
title_contains = Post.objects.filter(title__icontains='django')  # case-insensitive contains
recent = Post.objects.filter(created_at__year=2026)
long_posts = Post.objects.filter(views_count__gt=100)              # greater than
chained = Post.objects.filter(is_published=True).order_by('-created_at')[:5]  # top 5
The double-underscore syntax (title__icontains) is how Django ORM expresses SQL operators in Python. __gt/__lt/__gte/__lte = greater/less than (or equal). Slicing a QuerySet like [:5] works like a Python list, but translates to a SQL LIMIT clause — much more efficient than fetching everything and slicing in Python.
Update & delete shell
# UPDATE — single object
post = Post.objects.get(id=1)
post.title = "Updated Title"
post.save()

# UPDATE — bulk update matching many rows at once (faster, no .save() needed)
Post.objects.filter(is_published=False).update(is_published=True)

# DELETE — single object
post = Post.objects.get(id=2)
post.delete()

# DELETE — bulk delete
Post.objects.filter(views_count=0).delete()
.update() called directly on a QuerySet runs one efficient SQL UPDATE for all matching rows — much faster than looping through objects and calling .save() on each individually.

Day 10 — Django Forms

Today you will learn

  • Creating a plain forms.Form
  • Rendering a form in a template and handling submission
  • Built-in validation and CSRF protection

Theory

Django's forms module handles rendering HTML form fields, validating submitted data, and displaying error messages — all from one Python class definition. This is separate from ModelForms (Day 11), which are auto-generated directly from a model.

Every form that changes data (POST request) needs CSRF protection — a hidden token Django checks to make sure the submission actually came from your own site, not a malicious external page.

Defining a form blog/forms.py (new file)
from django import forms

class ContactForm(forms.Form):
    name = forms.CharField(max_length=100)
    email = forms.EmailField()
    message = forms.CharField(widget=forms.Textarea)
Handling it in a view blog/views.py
from django.shortcuts import render
from .forms import ContactForm

def contact(request):
    if request.method == 'POST':
        form = ContactForm(request.POST)
        if form.is_valid():
            name = form.cleaned_data['name']
            email = form.cleaned_data['email']
            # here you'd typically save to DB or send an email
            return render(request, 'blog/thanks.html', {'name': name})
    else:
        form = ContactForm()
    return render(request, 'blog/contact.html', {'form': form})
request.method == 'POST' checks whether the form was submitted (POST) or the page was just loaded (GET). form.is_valid() runs all field validation (required fields, correct email format, etc.) and returns True only if everything passes. form.cleaned_data is a dict of the validated, type-converted values — safe to use directly.
Template blog/templates/blog/contact.html
{% extends 'blog/base.html' %}
{% block content %}
<form method="post">
    {% csrf_token %}
    {{ form.as_p }}
    <button type="submit">Send</button>
</form>
{% endblock %}
{% csrf_token %} is mandatory on every form using POST — Django will reject the submission with a 403 error without it. {{ form.as_p }} renders every field wrapped in a <p> tag automatically, including error messages if validation failed.

Day 11 — ModelForms

Today you will learn

  • Generating a form directly from a model
  • Saving submitted data straight into the database

Theory

A ModelForm is a shortcut that auto-generates form fields from a model's fields — you don't repeat the field definitions like you did in ContactForm. It also gives you a .save() method that writes the form data directly to the database, tying models, forms, and views together neatly.

Defining a ModelForm blog/forms.py
from django import forms
from .models import Post

class PostForm(forms.ModelForm):
    class Meta:
        model = Post
        fields = ['title', 'content', 'is_published']   # or fields = '__all__'
The inner Meta class tells Django which model to base the form on and which fields to include. Using an explicit list (rather than '__all__') is safer — it stops new model fields from automatically becoming editable in the form without you noticing.
View using the ModelForm blog/views.py
from django.shortcuts import render, redirect
from .forms import PostForm

def create_post(request):
    if request.method == 'POST':
        form = PostForm(request.POST)
        if form.is_valid():
            form.save()                 # writes directly to the Post table
            return redirect('home')
    else:
        form = PostForm()
    return render(request, 'blog/post_form.html', {'form': form})
redirect('home') sends the browser to the URL named 'home' after a successful save — this follows the Post/Redirect/Get pattern, preventing the browser from re-submitting the same form if the user refreshes the page.

Day 12 — Class-Based Views: List & Detail

Today you will learn

  • Why Class-Based Views (CBVs) exist
  • Using ListView and DetailView for common patterns

Theory

Class-Based Views package common view patterns (list a model's records, show one record's detail, handle a form) into reusable classes so you write far less repetitive code than function-based views. Django provides "generic" CBVs for the most common jobs — ListView, DetailView, CreateView, UpdateView, DeleteView — which we'll cover across today and tomorrow.

ListView blog/views.py
from django.views.generic import ListView
from .models import Post

class PostListView(ListView):
    model = Post
    template_name = 'blog/post_list.html'   # default would be blog/post_list.html anyway
    context_object_name = 'posts'            # default would be 'object_list'
    ordering = ['-created_at']
Behind the scenes, ListView runs Post.objects.all(), orders it, and passes it to the template under the name given in context_object_name — equivalent to writing the function-based view from Day 4 manually, but in far fewer lines.
DetailView blog/views.py
from django.views.generic import DetailView

class PostDetailView(DetailView):
    model = Post
    template_name = 'blog/post_detail.html'
    context_object_name = 'post'
Wiring CBVs into urls.py blog/urls.py
from django.urls import path
from .views import PostListView, PostDetailView

urlpatterns = [
    path('', PostListView.as_view(), name='home'),
    path('post/<int:pk>/', PostDetailView.as_view(), name='post_detail'),
]
.as_view() converts the class into something Django's URL dispatcher can call like a function. <int:pk> is a URL path converter — it captures a number from the URL and passes it as pk (primary key), which DetailView automatically uses to fetch the matching object via get_object_or_404.
Detail template blog/templates/blog/post_detail.html
{% extends 'blog/base.html' %}
{% block content %}
    <h1>{{ post.title }}</h1>
    <p>{{ post.content }}</p>
    <small>Published {{ post.created_at }}</small>
{% endblock %}

Day 13 — CBVs: Create, Update, Delete

Today you will learn

  • Building full CRUD with generic CBVs
  • success_url, get_absolute_url, and confirmation pages

Theory

CreateView and UpdateView both combine a ModelForm with a template and a save step, covering what we hand-wrote in Day 11. DeleteView shows a confirmation page before deleting — you should never delete on a plain GET request, since browsers/crawlers can trigger GETs unexpectedly.

Full CRUD views blog/views.py
from django.views.generic import CreateView, UpdateView, DeleteView
from django.urls import reverse_lazy
from .models import Post

class PostCreateView(CreateView):
    model = Post
    fields = ['title', 'content', 'is_published']
    template_name = 'blog/post_form.html'
    success_url = reverse_lazy('home')

class PostUpdateView(UpdateView):
    model = Post
    fields = ['title', 'content', 'is_published']
    template_name = 'blog/post_form.html'
    success_url = reverse_lazy('home')

class PostDeleteView(DeleteView):
    model = Post
    template_name = 'blog/post_confirm_delete.html'
    success_url = reverse_lazy('home')
reverse_lazy() is used instead of redirect() here because these are class attributes, evaluated when the module loads (before urls.py is fully ready) — "lazy" defers the URL lookup until it's actually needed. CreateView and UpdateView can reuse the same template, since both just render a form.
URLs for full CRUD blog/urls.py
from .views import PostCreateView, PostUpdateView, PostDeleteView

urlpatterns += [
    path('post/new/', PostCreateView.as_view(), name='post_create'),
    path('post/<int:pk>/edit/', PostUpdateView.as_view(), name='post_edit'),
    path('post/<int:pk>/delete/', PostDeleteView.as_view(), name='post_delete'),
]
Delete confirmation template blog/templates/blog/post_confirm_delete.html
{% extends 'blog/base.html' %}
{% block content %}
    <h2>Delete "{{ post.title }}"?</h2>
    <form method="post">
        {% csrf_token %}
        <button type="submit">Yes, delete</button>
        <a href="{% url 'home' %}">Cancel</a>
    </form>
{% endblock %}

Day 14 — Model Relationships

Today you will learn

  • ForeignKey (many-to-one), ManyToManyField, and OneToOneField
  • Querying across relationships

Theory

  • ForeignKey — many-to-one. Many Comments can belong to one Post.
  • ManyToManyField — many-to-many. A Post can have many Tags, and a Tag can belong to many Posts.
  • OneToOneField — exactly one-to-one. A User has exactly one Profile.

on_delete is required on every ForeignKey — it decides what happens to related rows when the referenced row is deleted. CASCADE (delete related rows too) is the most common choice.

Adding relationships blog/models.py
from django.db import models
from django.contrib.auth.models import User

class Category(models.Model):
    name = models.CharField(max_length=50)
    def __str__(self):
        return self.name

class Post(models.Model):
    title = models.CharField(max_length=200)
    content = models.TextField()
    created_at = models.DateTimeField(auto_now_add=True)
    is_published = models.BooleanField(default=True)
    author = models.ForeignKey(User, on_delete=models.CASCADE, related_name='posts')
    categories = models.ManyToManyField(Category, blank=True)

    def __str__(self):
        return self.title

class Comment(models.Model):
    post = models.ForeignKey(Post, on_delete=models.CASCADE, related_name='comments')
    name = models.CharField(max_length=80)
    body = models.TextField()
    created_at = models.DateTimeField(auto_now_add=True)

    def __str__(self):
        return f"Comment by {self.name} on {self.post}"
related_name='comments' lets you go backwards from a Post to its Comments using post.comments.all() — without it, Django defaults to post.comment_set.all(), which works but is less readable.
Querying across relationships shell
post = Post.objects.get(id=1)

# forward relationship
print(post.author.username)         # ForeignKey -> author's username
print(post.categories.all())         # ManyToMany -> all categories on this post

# backward relationship (thanks to related_name)
print(post.comments.all())            # all comments on this post

# filtering through a relationship with double underscore
django_posts = Post.objects.filter(categories__name='Django')
alice_posts = Post.objects.filter(author__username='alice')
The double-underscore lookup syntax works across relationships too, not just field types — categories__name reaches "through" the ManyToMany field into the related Category model's name field.

Day 15 — User Authentication

Today you will learn

  • Django's built-in User model and django.contrib.auth
  • Registration, login, and logout views

Theory

Django ships with a complete authentication system: a User model, password hashing, session management, and ready-made views for login/logout. You rarely need to build these from scratch — mostly you wire up URLs and templates for the built-in views.

Registration view (custom) blog/views.py
from django.contrib.auth.forms import UserCreationForm
from django.shortcuts import render, redirect

def register(request):
    if request.method == 'POST':
        form = UserCreationForm(request.POST)
        if form.is_valid():
            form.save()
            return redirect('login')
    else:
        form = UserCreationForm()
    return render(request, 'blog/register.html', {'form': form})
UserCreationForm is a built-in ModelForm for the User model — it already handles password confirmation and hashing (passwords are never stored as plain text; Django hashes them with PBKDF2 by default).
Login & logout (built-in views) mysite/urls.py
from django.contrib.auth import views as auth_views

urlpatterns += [
    path('login/', auth_views.LoginView.as_view(template_name='blog/login.html'), name='login'),
    path('logout/', auth_views.LogoutView.as_view(), name='logout'),
]
Django provides LoginView and LogoutView ready to use — you just supply a login template; logout needs no template by default (it just clears the session and redirects).
Login template blog/templates/blog/login.html
{% extends 'blog/base.html' %}
{% block content %}
<form method="post">
    {% csrf_token %}
    {{ form.as_p }}
    <button type="submit">Log in</button>
</form>
{% endblock %}
Checking login state in a template any template
{% if user.is_authenticated %}
    <p>Welcome, {{ user.username }}! <a href="{% url 'logout' %}">Logout</a></p>
{% else %}
    <a href="{% url 'login' %}">Login</a> | <a href="{% url 'register' %}">Register</a>
{% endif %}
user is automatically available in every template (via Django's built-in context processor) — user.is_authenticated is False for anonymous visitors and True once logged in.

Day 16 — Authorization & Permissions

Today you will learn

  • Restricting views to logged-in users
  • Restricting edit/delete to only the post's owner

Theory

Authentication answers "who are you?"; authorization answers "are you allowed to do this?". Django provides both a decorator (for function-based views) and a mixin (for class-based views) to require login, plus manual checks for object-level permissions like "only the owner can edit their own post".

Function-based view protection blog/views.py
from django.contrib.auth.decorators import login_required

@login_required
def create_post(request):
    # ... only reachable by logged-in users; others get redirected to LOGIN_URL
    ...
Class-based view protection blog/views.py
from django.contrib.auth.mixins import LoginRequiredMixin, UserPassesTestMixin
from django.views.generic import CreateView, UpdateView

class PostCreateView(LoginRequiredMixin, CreateView):
    model = Post
    fields = ['title', 'content', 'is_published']
    login_url = 'login'

    def form_valid(self, form):
        form.instance.author = self.request.user   # auto-assign the logged-in user
        return super().form_valid(form)

class PostUpdateView(LoginRequiredMixin, UserPassesTestMixin, UpdateView):
    model = Post
    fields = ['title', 'content', 'is_published']

    def test_func(self):
        post = self.get_object()
        return self.request.user == post.author   # only the owner can edit
Mixins are added as extra parent classes — Python checks them left to right, so LoginRequiredMixin runs before UserPassesTestMixin. form_valid() is called automatically after validation passes, right before saving — the perfect hook to attach the current user as the author. test_func() must return True/False; UserPassesTestMixin blocks access (403) if it returns False.

Day 17 — Django REST Framework: Serializers

Today you will learn

  • What DRF adds on top of Django, and why APIs matter
  • Converting model instances to/from JSON with serializers

Theory

Django REST Framework (DRF) is the standard toolkit for building JSON APIs on top of Django — used when your frontend is a separate app (React, mobile app) that needs data over HTTP instead of rendered HTML. A serializer is DRF's equivalent of a form: it converts complex data (model instances, querysets) into JSON, and validates/converts incoming JSON back into Python objects.

Install & configure terminal / settings.py
pip install djangorestframework
INSTALLED_APPS = [
    # ...
    'rest_framework',
    'blog',
]
Serializer blog/serializers.py (new file)
from rest_framework import serializers
from .models import Post

class PostSerializer(serializers.ModelSerializer):
    author = serializers.ReadOnlyField(source='author.username')

    class Meta:
        model = Post
        fields = ['id', 'title', 'content', 'author', 'is_published', 'created_at']
ModelSerializer is the DRF equivalent of ModelForm — it auto-generates fields from the model. ReadOnlyField(source='author.username') exposes the author's username in the API output (instead of just their numeric ID) without allowing it to be edited through the API.
Using it manually shell
from blog.models import Post
from blog.serializers import PostSerializer

post = Post.objects.first()
serializer = PostSerializer(post)
print(serializer.data)          # a dict, ready to convert to JSON

posts = Post.objects.all()
many_serializer = PostSerializer(posts, many=True)   # many=True for querysets/lists
print(many_serializer.data)
many=True is required whenever serializing a queryset or list instead of a single object — it tells DRF to loop through and serialize each item.

Day 18 — DRF: Views & Routers

Today you will learn

  • Building a full CRUD API with ViewSet and Router
  • Testing the API with the browsable API interface

Theory

Just like Django has generic CBVs, DRF has ViewSets that bundle list/create/retrieve/update/delete logic for a model into one class. A Router then automatically generates all the URL patterns for that ViewSet — no manual path() entries needed for each CRUD action.

ViewSet blog/views.py
from rest_framework import viewsets, permissions
from .models import Post
from .serializers import PostSerializer

class PostViewSet(viewsets.ModelViewSet):
    queryset = Post.objects.all().order_by('-created_at')
    serializer_class = PostSerializer
    permission_classes = [permissions.IsAuthenticatedOrReadOnly]

    def perform_create(self, serializer):
        serializer.save(author=self.request.user)
ModelViewSet gives you list, create, retrieve, update, partial_update, and destroy — the entire CRUD API — in one class. IsAuthenticatedOrReadOnly means anyone can GET (read), but only logged-in users can POST/PUT/DELETE. perform_create() is DRF's version of form_valid() — the hook to attach the current user before saving.
Router-based URLs mysite/urls.py
from rest_framework.routers import DefaultRouter
from blog.views import PostViewSet

router = DefaultRouter()
router.register('api/posts', PostViewSet, basename='post')

urlpatterns += router.urls
This single router.register() call automatically creates: GET/POST /api/posts/ (list & create) and GET/PUT/PATCH/DELETE /api/posts/<id>/ (retrieve, update, delete) — 6 endpoints from one line.
Testing the API browser / requests
import requests

# GET all posts
response = requests.get("http://127.0.0.1:8000/api/posts/")
print(response.json())

# POST a new post (requires authentication in a real setup)
new_post = {"title": "API Post", "content": "Created via DRF", "is_published": True}
response = requests.post("http://127.0.0.1:8000/api/posts/", json=new_post)
print(response.status_code, response.json())
Visiting /api/posts/ directly in a browser shows DRF's browsable API — an interactive HTML page with forms for testing every endpoint without any extra tools, which is one of DRF's most loved features for development.

Day 20 — Deployment Basics & Capstone Project

Today you will learn

  • Preparing a Django project for production
  • Environment variables and requirements.txt
  • Pulling everything together into one finished Blog application

Theory

Moving from development to a live server requires several safety and performance changes. Never deploy with development settings — they're insecure and slow for real traffic.

  • DEBUG = False — turns off Django's detailed error pages, which can leak sensitive info
  • ALLOWED_HOSTS — a list of domain names allowed to serve this project (prevents HTTP Host header attacks)
  • Secret key & passwords should never be hardcoded — load them from environment variables
  • collectstatic — gathers all static files into one folder for the production web server to serve directly
  • WSGI/ASGI server — Django's own runserver is for development only; production uses something like Gunicorn behind Nginx
Production-ready settings mysite/settings.py
import os

DEBUG = os.environ.get('DJANGO_DEBUG', 'False') == 'True'
SECRET_KEY = os.environ.get('DJANGO_SECRET_KEY')
ALLOWED_HOSTS = os.environ.get('DJANGO_ALLOWED_HOSTS', '').split(',')

STATIC_ROOT = BASE_DIR / 'staticfiles'    # target folder for collectstatic
os.environ.get() reads values from environment variables set on the server — this way your actual secret key and database password never appear in your code or get pushed to GitHub.
Preparing for deployment terminal
# freeze installed packages so the server can install the same versions
pip freeze > requirements.txt

# gather all static files into STATIC_ROOT
python manage.py collectstatic

# install a production WSGI server
pip install gunicorn
gunicorn mysite.wsgi:application
requirements.txt lists every package and exact version your project needs — hosting platforms (Render, Railway, PythonAnywhere, Heroku) read this file to rebuild your environment automatically during deployment.
