HTML & CSS for Software Testing in Selenium (Python)

Everything a tester needs to know about HTML structure and CSS to confidently locate, interact with, and validate web elements using Selenium WebDriver.

1. HTML Document Structure

Every HTML page is a tree of nested elements (tags). Selenium locates elements by walking this tree — understanding the structure is the foundation of every locator strategy.

<html>
  <head>
    <title>Page Title</title>
  </head>
  <body>
    <div id="app">
      <h1>Welcome</h1>
    </div>
  </body>
</html>

Key terms a tester must know:

TermMeaning
Element / TagA building block like <div>, <button>, <input>
AttributeExtra info on a tag: id, class, name, type, href
Parent / ChildNesting relationship — used heavily in XPath (parent::, child::)
SiblingElements at the same level under the same parent
DOMDocument Object Model — the in-memory tree Selenium actually queries

2. Attributes Selenium Cares About

AttributeUsed ForSelenium Example
idUnique identifier (best locator when stable)driver.find_element(By.ID, "username")
nameCommon on form fields, used in form submissiondriver.find_element(By.NAME, "email")
classStyling hook, can repeat on multiple elementsdriver.find_elements(By.CLASS_NAME, "card")
typeDistinguishes input variants (text, checkbox, submit...)driver.find_element(By.CSS_SELECTOR, "input[type='email']")
hrefLink destination on anchor tagsdriver.find_element(By.PARTIAL_LINK_TEXT, "Download")
placeholderHint text inside inputs, useful as a fallback locatorinput[placeholder='Search']
data-*Custom attributes, increasingly the most stable test hooks[data-testid='submit-btn']
disabled / readonly / checked / selectedBoolean state attributes, used to assert element stateelement.get_attribute("disabled")
💡 Tip: If a site has data-testid attributes, always prefer them — they're added specifically for automation and rarely change, unlike CSS classes used for styling.

3. Form Elements

Forms are the most-tested part of any application: login, registration, search, checkout. Know each input type and the Selenium action it expects.

HTMLPurposeSelenium Action
<input type="text">Free text entry.send_keys("value")
<input type="password">Masked text entry.send_keys("secret")
<input type="checkbox">Boolean toggle.click(), check with .is_selected()
<input type="radio">Single choice from a group.click()
<select><option>DropdownSelect(element).select_by_visible_text("Python")
<select multiple>Multi-select dropdownSelect(element).select_by_index(n) repeated
<textarea>Multi-line text.send_keys("text")
<input type="file">File upload.send_keys("/path/to/file") (no click needed)
<input type="date">Date picker.send_keys("06/30/2026") (format varies by browser)
<button> / <input type="submit">Form submission / actions.click()
<label for="id">Links text to a field via the for/id matchUseful for XPath: //label[text()='Email']/following-sibling::input

4. Tables

Tables (<table>, <tr>, <td>, <th>) appear constantly in dashboards, reports, and search results. Testers frequently need to read rows/columns, search for a value, or validate cell content.

<table>
  <tr><th>Name</th><th>Role</th></tr>
  <tr><td>John</td><td>QA</td></tr>
</table>

Common Python/Selenium pattern

# loop all rows, then all cells in each row
rows = driver.find_elements(By.CSS_SELECTOR, "table tr")
for row in rows:
    cells = row.find_elements(By.TAG_NAME, "td")
    values = [c.text for c in cells]
    print(values)

XPath patterns for tables

// find a cell by its row's text (find "John"'s row, then get the Role cell)
//td[text()='John']/following-sibling::td[1]

// get the 2nd row's 3rd cell
//table/tr[2]/td[3]

HTMLSelenium Locator

<a href="#">Download Selenium Guide</a>By.PARTIAL_LINK_TEXT, "Download"

💡 Link text locators only work on <a> tags, not buttons or divs styled to look like links.

6. Semantic / Layout Tags

Modern apps use meaningful tags instead of generic <div> soup. Knowing these helps you read a page's structure quickly when designing locators.

TagMeaning
<header>Top banner / page title area
<nav>Navigation menu links
<main>Primary page content (one per page)
<section>A thematic grouping of content
<article>Self-contained content block (a card, a post)
<aside>Sidebar or tangential content
<footer>Bottom of page / section
<div> / <span>Generic containers with no semantic meaning — rely entirely on id/class

7. iFrames, Windows & Alerts

These don't have unique HTML tags you "style," but they change how Selenium must reach an element.

ConceptHTML CauseSelenium Handling
iFrame<iframe id="frame1">driver.switch_to.frame("frame1") ... driver.switch_to.default_content()
New Tab/Window<a target="_blank">driver.switch_to.window(driver.window_handles[-1])
JS Alert/Confirm/Promptonclick="alert(...)"driver.switch_to.alert.accept() / .dismiss() / .send_keys()
⚠️ JS alerts are not part of the DOM — you cannot use find_element on them. Always switch to driver.switch_to.alert first.

8. Dynamic / Hidden Elements

Real apps load content asynchronously or hide it conditionally. This is where most flaky tests come from.

<div style="display:none">Hidden Content</div>
StateWhat it meansSelenium check
display:noneRemoved from layout, not clickableelement.is_displayed() → False
visibility:hiddenInvisible but still takes up spaceelement.is_displayed() → False
Not yet renderedElement doesn't exist in DOM yet (AJAX-loaded)Use WebDriverWait + expected_conditions
Disabled<button disabled>element.is_enabled() → False

Standard explicit wait pattern

from selenium.webdriver.support.ui import WebDriverWait
from selenium.webdriver.support import expected_conditions as EC
from selenium.webdriver.common.by import By

el = WebDriverWait(driver, 10).until(
    EC.visibility_of_element_located((By.ID, "hidden-content"))
)

9. CSS Syntax Basics

selector {
  property: value;
}

/* example */
.login-btn {
  background-color: blue;
  color: white;
}

You don't need to write CSS as a tester — but you must read CSS selectors fluently, because By.CSS_SELECTOR uses the exact same syntax as a stylesheet.

Pseudo-classUse

13. Attribute Selectors (Deep Dive)

PatternMatches
[type]Any element that has a type attribute (any value)

[class*='card']Class contains "card" anywhere
[data-testid='submit']Custom data attribute exact match (best practice locator)

14. Locator Strategy: Order of Preference

  1. id — fastest, most reliable, use first if present and stable
  2. data-testid / data-* attributes — built for testing, very stable
  3. name — reliable on form fields
  4. CSS selector — fast, readable, supports almost everything except "text contains"
  5. link text / partial link text — only for anchor tags
  6. XPath — most powerful (can match by text, traverse up/down the tree) but slowest and most brittle to layout changes
  7. class name — risky alone, since classes are often shared by many elements and reused for styling
💡 Rule of thumb: reach for CSS selectors by default, and only drop to XPath when you need to match by visible text or traverse to a parent/ancestor — CSS cannot select a parent based on its child.

16. XPath Cheat Sheet

XPathMeaning
//tag[@attr='value']Element with exact attribute match
//tag[contains(@attr,'val')]Attribute contains a substring
//tag[text()='Login']Exact visible text match
//tag[contains(text(),'Log')]Partial visible text match
//div[@class='card']//buttonAny-depth descendant button inside a card div
//input/following-sibling::spanThe span right after an input
//label[text()='Email']/following::input[1]First input appearing after a label (when no for attribute links them)
(//table//tr)[2]2nd row in the (first) matched table
//*[@id='username']Wildcard tag with a given id (rarely needed — prefer By.ID directly)

18. Quick Summary — What to Actually Memorize

HTML Must-Knows

  • Reading nested tags (parent/child/sibling)
  • id, class, name, type, data-* attributes
  • All form input types and their Selenium actions
  • Table structure: tr / td / th
  • Hidden vs disabled vs not-yet-rendered states
  • iframe and target="_blank" implications

CSS Must-Knows

  • #id, .class, tag basics
  • Descendant ( ) vs direct child (>)
  • Attribute selectors: [attr='val'], ^=, $=, *=
  • :nth-child() for tables/lists
  • When CSS can't do it (text match, parent traversal) → switch to XPath

Once these are comfortable, the best next step is hands-on repetition: open the practice playground page and locate every element using at least two different strategies (e.g., ID and CSS, or CSS and XPath) until it feels automatic.
