Day 1 — Introduction & Setup

Today you will learn

  • What a Database is and how it differs from file systems
  • DBMS vs Relational DBMS (RDBMS)
  • Understanding SQL categories: DDL, DML, DCL, and TCL
  • Setting up a local database server (PostgreSQL or MySQL)

Theory

A Database is an organized collection of structured data, typically stored and accessed electronically. Unlike flat text files or spreadsheets, databases handle massive scale, enforce strict integrity rules, support multiple concurrent users, and guarantee data safety.

DBMS vs RDBMS

  • DBMS (Database Management System) — Software to store and manage data (flat files, XML, NoSQL documents).
  • RDBMS (Relational DBMS) — Stores data in tabular formats (rows and columns) linked together by relationships. Relational systems are built on set theory and use SQL (Structured Query Language) as their query language. Examples: PostgreSQL, MySQL, SQLite, Oracle, SQL Server.

Categories of SQL Commands

CategoryFull NamePurposeCommands

DMLData Manipulation LanguageManipulate data stored in tablesSELECT, INSERT, UPDATE, DELETE
DCLData Control LanguageControl access permissionsGRANT, REVOKE
TCLTransaction Control LanguageManage changes made by DMLCOMMIT, ROLLBACK, SAVEPOINT
Installing PostgreSQL local config

To run SQL queries, it is recommended to install a local client and server. For these 20 days, we'll write standard SQL which works across PostgreSQL, MySQL, and SQLite.

  • Set a password during installation (write it down, you will need it).

Day 2 — The SELECT Statement

Today you will learn

  • The basic syntax of a database query
  • Selecting specific columns and querying all columns with asterisk (*)
  • Using arithmetic expressions in query columns
  • Naming column aliases with `AS`

Theory

The SELECT statement is the most common command in SQL. It is a DML operation used to retrieve data from one or more database tables. The RDBMS returns the results in a grid of rows and columns called a result set.

Syntax Structure

The simplest query requires two clauses:

  • SELECT — specifies which columns to retrieve.
  • FROM — specifies the table containing the columns.

Arithmetic & Column Aliases

You can perform calculations (addition, subtraction, multiplication, division) directly in the SELECT clause. A column alias renaming (using AS) changes the title displayed in the result grid temporarily. It is highly recommended to make calculations readable.

Basic queries queries
-- 1. Query all columns from the employees table
SELECT * FROM employees;

-- 2. Query only specific columns
SELECT first_name, last_name, salary FROM employees;

-- 3. Arithmetic operations in queries (calculate annual salary)
SELECT first_name, last_name, salary, salary * 12 AS annual_salary 
FROM employees;

-- 4. Concatenating strings (PostgreSQL syntax)
SELECT first_name || ' ' || last_name AS full_name 
FROM employees;
Using * is convenient during database testing, but in production applications, you should always request specific columns to save memory and network bandwidth.

Day 3 — Filtering Data (WHERE Clause)

Today you will learn

  • How to filter results using the WHERE clause
  • Using comparison operators: =, <>, <, >, <=, >=
  • Logical operators: AND, OR, NOT
  • Special operators: BETWEEN, IN, LIKE, IS NULL

Theory

By default, queries return all rows from a table. To restrict rows, we add a WHERE clause. The WHERE clause contains one or more conditions. If a row satisfies the condition (it evaluates to True), it is included in the result set; otherwise, it is ignored.

Special Operators

  • BETWEEN a AND b — Filters values within an inclusive range.
  • IN (val1, val2, ...) — Matches any value in a list.
  • LIKE — Performs pattern matching using wildcards: % (zero or more characters) and _ (exactly one character).
  • IS NULL — Checks for empty or undefined values (you cannot use = NULL!).
Filtering examples queries
-- 1. Filter by numeric comparison
SELECT * FROM employees WHERE salary > 50000;

-- 2. Combine filters with logical operators
SELECT * FROM employees 
WHERE department_id = 5 AND salary >= 40000;

-- 3. Find employees whose salary is between 30k and 60k (inclusive)
SELECT * FROM employees 
WHERE salary BETWEEN 30000 AND 60000;

-- 4. Find employees in specific departments
SELECT * FROM employees 
WHERE department_id IN (1, 3, 5);

-- 5. Pattern matching (names starting with 'J')
SELECT * FROM employees 
WHERE first_name LIKE 'J%';

-- 6. Find records with missing data
SELECT * FROM employees 
WHERE manager_id IS NULL;

Day 4 — Sorting & Limiting Data

Today you will learn

  • Sorting query results with ORDER BY
  • Ascending (ASC) vs Descending (DESC) sort order
  • Restricting result counts with LIMIT
  • Skipping rows using OFFSET (paging queries)

Theory

Relational databases do not store rows in a guaranteed order. If you query a table, rows are returned in whatever order the database engine finds them. To enforce a sort order, we use ORDER BY.

Limit and Offset

When dealing with tables containing millions of rows, returning everything crashes applications. LIMIT restricts the output count (e.g. top 10). Combining LIMIT with OFFSET allows you to create pagination (e.g., page 2 has limit 10, offset 10).

Sorting and Paging queries
-- 1. Sort by salary ascending (default)
SELECT * FROM employees 
ORDER BY salary;

-- 2. Sort by salary descending
SELECT * FROM employees 
ORDER BY salary DESC;

-- 3. Sort by multiple columns (department first, then salary descending)
SELECT * FROM employees 
ORDER BY department_id ASC, salary DESC;

-- 4. Get the top 5 highest-paid employees
SELECT * FROM employees 
ORDER BY salary DESC 
LIMIT 5;

-- 5. Paging: Get rows 11 to 20 (page 2)
SELECT * FROM employees 
ORDER BY employee_id 
LIMIT 10 OFFSET 10;

Day 5 — Aggregate Functions

Today you will learn

  • Understanding aggregation in SQL
  • Count records using COUNT()
  • Calculate sums and averages using SUM() and AVG()
  • Find boundaries using MIN() and MAX()
  • How NULL values behave in aggregate computations

Theory

Aggregate functions perform a calculation on a set of values in a column and return a single summary value. Unlike arithmetic operations which work row-by-row, aggregates summarize data vertically across multiple rows.

Aggregate Behavior with NULLs

Most aggregate functions ignore NULL values. For instance, AVG(salary) computes the average of only the rows where salary is NOT null. The exception is COUNT(*), which counts every row matching filters regardless of column null values.

Aggregation Examples queries
-- 1. Count all rows in employees
SELECT COUNT(*) AS total_employees FROM employees;

-- 2. Count non-null values (employees with a manager)
SELECT COUNT(manager_id) AS managed_employees FROM employees;

-- 3. Find salary statistics
SELECT SUM(salary) AS total_payroll,
       AVG(salary) AS average_salary,
       MIN(salary) AS lowest_salary,
       MAX(salary) AS highest_salary
FROM employees;

-- 4. Distinct counts (number of unique departments represented)
SELECT COUNT(DISTINCT department_id) AS active_departments FROM employees;

Day 6 — Grouping Data (GROUP BY)

Today you will learn

  • Split query data into category buckets using GROUP BY
  • Rules of select items in grouped queries
  • Filtering grouped results using HAVING
  • The difference between WHERE and HAVING filters

Theory

The GROUP BY clause is used in collaboration with aggregate functions to group the result set by one or more columns. For instance, instead of finding the average salary of the entire company, we can find the average salary per department.

Critical Rule of Grouping

When using GROUP BY, every column selected in your query must either be listed in the GROUP BY clause, or wrapped inside an aggregate function. Selecting columns outside this raises a syntax error.

WHERE vs HAVING

  • WHERE — Filters individual rows before groups are calculated. It cannot contain aggregate functions.
  • HAVING — Filters grouped summary rows after groups are calculated. It filters on aggregated metrics.
Grouping queries queries
-- 1. Calculate average salary per department
SELECT department_id, AVG(salary) AS avg_sal 
FROM employees 
GROUP BY department_id;

-- 2. Count employees in each job role, sorted by employee count descending
SELECT job_title, COUNT(*) AS emp_count 
FROM employees 
GROUP BY job_title 
ORDER BY emp_count DESC;

-- 3. Filter groups (find departments with average salary above 50,000)
SELECT department_id, AVG(salary) AS avg_sal 
FROM employees 
GROUP BY department_id 
HAVING AVG(salary) > 50000;

-- 4. Using WHERE and HAVING together
-- (Exclude low-tier employees, then group, then filter small departments)
SELECT department_id, COUNT(*) AS emp_count, AVG(salary) AS avg_sal
FROM employees
WHERE salary > 30000
GROUP BY department_id
HAVING COUNT(*) >= 2;

Day 7 — Joins Part 1: Inner Joins

Today you will learn

  • What Joins are and how they link tables
  • Understanding primary key vs foreign key matching
  • Combining tables with INNER JOIN
  • Preventing name collisions with table aliases

Theory

A relational database splits data into normalized tables. To rebuild a full view, we use joins. A join operation links rows from two tables based on a common matching column — typically a Primary Key in one table and a corresponding Foreign Key in the other.

Inner Join

An INNER JOIN returns rows only when there is a match in both tables. If a row in the first table does not have a matching ID in the second table, it is completely excluded from the result set.

Inner Join Syntax queries
-- 1. Combine employees and departments tables
SELECT e.employee_id, e.first_name, e.last_name, d.department_name
FROM employees AS e
INNER JOIN departments AS d 
ON e.department_id = d.department_id;

-- 2. Join three tables (employees, departments, locations)
SELECT e.first_name, e.last_name, d.department_name, l.city
FROM employees e
INNER JOIN departments d ON e.department_id = d.department_id
INNER JOIN locations l ON d.location_id = l.location_id;
Using table aliases like e and d is standard practice. It keeps joins readable, and is mandatory when both tables share matching column names (e.g. department_id).

Day 8 — Joins Part 2: Outer Joins & More

Today you will learn

  • Understanding outer join dynamics
  • Keep un-matched rows with LEFT JOIN and RIGHT JOIN
  • Comprehensive combinations using FULL JOIN
  • Cross Joins and Self Joins

Theory

Inner Joins discard rows that fail to match. Sometimes, we want to retain those un-matched records (for example, displaying a list of all products, even those that have never been ordered). This is what outer joins do.

  • LEFT JOIN — Returns all rows from the left table, and matched rows from the right. If no match exists, NULLs are inserted for right-table columns.
  • RIGHT JOIN — Returns all rows from the right table, and matched rows from the left. (Equivalent to a Left Join if you swap the table order).
  • FULL JOIN — Returns rows when there is a match in either left or right tables, filling missing fields with NULLs.
  • CROSS JOIN — Returns the Cartesian product of both tables (every row in A paired with every row in B).
  • SELF JOIN — Joins a table to itself (useful for hierarchical structures, like matching employees to their managers inside the same table).
Outer and Self Joins queries
-- 1. Left Join: List all departments, showing employee counts (even empty depts)
SELECT d.department_name, COUNT(e.employee_id) AS total_staff
FROM departments d
LEFT JOIN employees e ON d.department_id = e.department_id
GROUP BY d.department_name;

-- 2. Self Join: Find employee names alongside their manager's name
SELECT e.first_name AS employee, m.first_name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.employee_id;

-- 3. Cross Join: Generate all combinations of sizes and colors
SELECT s.size_name, c.color_name
FROM sizes s
CROSS JOIN colors c;
A Self Join requires giving the same table two different aliases (e.g. e for employee role, m for manager role) so the query parser treats them as separate entities.

Day 9 — Subqueries

Today you will learn

  • What a subquery is and where it can be placed
  • Single-row (scalar) subqueries
  • Multi-row subqueries using IN, ANY, and ALL
  • Correlated subqueries
  • Checking record existence with EXISTS

Theory

A Subquery (or nested query) is a query written inside another query's clauses. It acts as a dynamic source of data that feeds the outer query.

Subquery Types

  • Scalar Subquery — Returns exactly one value (one cell). Used with comparison operators like =, >.
  • Multi-Row Subquery — Returns a column of values. Used with IN, ANY, ALL.
  • Correlated Subquery — References a column from the outer query. The subquery runs repeatedly (once for every row processed by the outer query).
  • EXISTS / NOT EXISTS — Evaluates to True or False depending on whether the subquery returns any rows at all. Highly optimized for performance.
Subquery Examples queries
-- 1. Scalar Subquery: Find employees earning more than the company average
SELECT first_name, last_name, salary 
FROM employees 
WHERE salary > (SELECT AVG(salary) FROM employees);

-- 2. Multi-row Subquery: Find employees working in IT locations
SELECT first_name, last_name FROM employees 
WHERE department_id IN (
    SELECT department_id FROM departments 
    WHERE location_id = 1700
);

-- 3. Correlated Subquery: Find employees earning more than their department's average
SELECT e.first_name, e.salary, e.department_id
FROM employees e
WHERE e.salary > (
    SELECT AVG(salary) 
    FROM employees 
    WHERE department_id = e.department_id
);

-- 4. EXISTS: Find departments that have at least one employee earning over 10k
SELECT d.department_name FROM departments d
WHERE EXISTS (
    SELECT 1 FROM employees e 
    WHERE e.department_id = d.department_id AND e.salary > 10000
);

Day 10 — CTEs (Common Table Expressions)

Today you will learn

  • What a Common Table Expression (CTE) is
  • Declaring CTEs using the WITH clause
  • Improving query readability and maintenance over subqueries
  • Defining multiple CTEs in a single query

Theory

A Common Table Expression (CTE) is a temporary result set defined within the scope of a single query. It works like a virtual table that exists only during the execution of that query.

Why use CTEs instead of Subqueries?

  • Readability — CTEs are defined at the top of the script and build logically, whereas subqueries nest deeply, which is harder to read.
  • Reusability — You can reference the same CTE multiple times within the query (e.g. self-joining temporary aggregations).
CTE Syntax queries
-- 1. Simple CTE: Calculate department averages, then query it
WITH dept_averages AS (
    SELECT department_id, AVG(salary) AS avg_sal
    FROM employees
    GROUP BY department_id
)
SELECT e.first_name, e.salary, da.avg_sal
FROM employees e
INNER JOIN dept_averages da ON e.department_id = da.department_id
WHERE e.salary > da.avg_sal;

-- 2. Defining Multiple CTEs
WITH regional_sales AS (
    SELECT region_id, SUM(amount) AS total_sales
    FROM sales
    GROUP BY region_id
),
top_regions AS (
    SELECT region_id FROM regional_sales
    WHERE total_sales > (SELECT AVG(total_sales) FROM regional_sales)
)
SELECT region_id FROM top_regions;

Day 11 — Creating & Modifying Tables (DDL)

Today you will learn

  • Designing tables with CREATE TABLE
  • Common SQL datatypes: INT, VARCHAR, DECIMAL, DATE, BOOLEAN
  • Modifying table schema using ALTER TABLE
  • Removing tables using DROP TABLE and TRUNCATE

Theory

Data Definition Language (DDL) commands are used to define, alter, and delete the physical structure of database schemas.

Common Data Types

  • INTEGER / INT — Whole numbers.
  • VARCHAR(length) — Variable character text up to a maximum length limit.
  • DECIMAL(precision, scale) — Exact numbers (e.g. DECIMAL(10, 2) stores numbers with up to 10 total digits, 2 after the decimal — ideal for currency).
  • DATE — Stores calendar dates (YYYY-MM-DD).
  • BOOLEAN — True or False flags.

Delete Commands Comparison

  • DELETE FROM table; — DML command. Removes rows one by one. Slow but can be rolled back.
  • TRUNCATE TABLE table; — DDL command. Fast operation that drops and recreates the empty structure. Cannot be rolled back.
  • DROP TABLE table; — DDL command. Deletes table data AND the table structural definition completely.
DDL Operations ddl
-- 1. Create a students table
CREATE TABLE students (
    student_id INT,
    first_name VARCHAR(50),
    last_name VARCHAR(50),
    gpa DECIMAL(3, 2),
    enrollment_date DATE,
    is_active BOOLEAN
);

-- 2. Add a new column to the table
ALTER TABLE students ADD COLUMN email VARCHAR(100);

-- 3. Drop a column
ALTER TABLE students DROP COLUMN is_active;

-- 4. Delete the table structure completely
DROP TABLE students;

Day 12 — Database Constraints

Today you will learn

  • Enforcing database integrity rules
  • Uniqueness and identity: PRIMARY KEY
  • Enforcing relationship integrity: FOREIGN KEY
  • Value constraints: NOT NULL, UNIQUE, CHECK
  • Setting default values with DEFAULT

Theory

Constraints are rules enforced by the RDBMS on table data columns to guarantee accuracy, validity, and consistency. If any insert or update statement violates a constraint, the database engine aborts the operation and returns an error.

Constraint Types

  • PRIMARY KEY — Uniquely identifies each row. Enforces uniqueness and NOT NULL implicitly. (Only one per table).
  • FOREIGN KEY — References a Primary Key in another table, preventing orphaned records.
  • UNIQUE — Guarantees all values in a column are distinct.
  • NOT NULL — Prevents storing empty NULL markers in this column.
  • CHECK (condition) — Restricts values to those meeting a boolean condition (e.g. price > 0).
  • DEFAULT value — Sets a default fallback value if none is specified during an insert.
Tables with constraints ddl
-- 1. Create lookup categories table
CREATE TABLE categories (
    category_id INT PRIMARY KEY,
    category_name VARCHAR(50) NOT NULL UNIQUE
);

-- 2. Create products table linked to categories
CREATE TABLE products (
    product_id INT PRIMARY KEY,
    product_name VARCHAR(100) NOT NULL,
    price DECIMAL(10, 2) DEFAULT 0.00,
    category_id INT,
    created_at DATE DEFAULT CURRENT_DATE,
    CONSTRAINT fk_category FOREIGN KEY (category_id) REFERENCES categories(category_id),
    CONSTRAINT chk_price CHECK (price >= 0)
);

Day 13 — Modifying Data (DML)

Today you will learn

  • Adding records using INSERT INTO
  • Updating columns using UPDATE
  • The danger of leaving out a WHERE clause in UPDATE/DELETE
  • Removing rows using DELETE

Theory

Data Manipulation Language (DML) commands write, alter, and delete individual rows within a database structure.

Insert Variants

You can insert a single row, insert multiple rows in a single batch statement, or copy rows dynamically from another table using a combined INSERT INTO ... SELECT statement.

The Dangerous WHERE Clause

When running UPDATE or DELETE, omitting the WHERE clause applies the modifications to every single row in the table. There is no simple undo in raw database clients, so double-check before running!

DML Operations dml
-- 1. Insert a single record
INSERT INTO categories (category_id, category_name) 
VALUES (1, 'Electronics');

-- 2. Insert multiple records at once
INSERT INTO categories (category_id, category_name) VALUES 
(2, 'Books'),
(3, 'Home & Kitchen');

-- 3. Update data (highly specific with WHERE)
UPDATE products 
SET price = price * 0.90 
WHERE category_id = 1;

-- 4. Delete specific records
DELETE FROM products 
WHERE product_id = 101;

Day 14 — SQL Views

Today you will learn

  • What a SQL View is (virtual tables)
  • Creating views using CREATE VIEW
  • Simplifying complex multi-table joins using views
  • Security benefits of using views for access control
  • Dropping views with DROP VIEW

Theory

A View is a virtual table whose contents are defined by a query. It does not store data physically on disk. Instead, every time a query requests rows from the view, the RDBMS runs the underlying query definition dynamically behind the scenes.

Use Cases

  • Simplification — Hide complex outer/inner joins, aggregations, and subqueries behind a simple virtual table name.
  • Security — Give external users read access to a view that hides sensitive columns (like salary or passwords) while protecting the main tables.
Views Implementation views
-- 1. Create a view that returns employees with department details
CREATE VIEW v_employee_details AS
SELECT e.employee_id, e.first_name, e.last_name, d.department_name, e.salary
FROM employees e
INNER JOIN departments d ON e.department_id = d.department_id;

-- 2. Query the view just like a normal table
SELECT first_name, department_name 
FROM v_employee_details 
WHERE salary > 50000;

-- 3. Delete the view definition
DROP VIEW v_employee_details;

Day 15 — Indexing & Performance Tuning

Today you will learn

  • How RDBMS searches for data: Full Table Scan vs Index Scan
  • Creating indexes with CREATE INDEX
  • How indexes speed up SELECT, but slow down INSERT/UPDATE
  • Analyzing execution plans using EXPLAIN

Theory

Without indexes, if a query requests a specific record (e.g. WHERE email = 'user@example.com'), the database must run a Full Table Scan — reading every single data block from the disk top-to-bottom. If you have 10 million rows, this is extremely slow.

What is an Index?

An Index is a separate data structure (typically a B-Tree) containing sorted pointers to rows. It allows the database engine to search for a key in logarithmic time (Index Scan), similar to looking up a word in a book index.

The Trade-off

Indexes speed up read queries (SELECT), but they slow down writes (INSERT, UPDATE, DELETE) because the database must update the index structure on disk every time rows change. Only index columns that are frequently filtered on (like email or IDs) or used in joins.

Indexes and Explain tuning
-- 1. Create an index on the email column of users table
CREATE INDEX idx_user_email ON users(email);

-- 2. Analyze query execution path (PostgreSQL / MySQL syntax)
EXPLAIN SELECT * FROM users WHERE email = 'test@example.com';

-- 3. Drop index
DROP INDEX idx_user_email;
Running EXPLAIN outputs the execution plan grid. Look for Seq Scan (sequential table scan, bad) vs Index Scan (indexed lookup, good) to evaluate optimization.

Day 16 — Window Functions

Today you will learn

  • Understanding Window Functions and how they differ from GROUP BY
  • The OVER clause with PARTITION BY and ORDER BY
  • Ranking functions: ROW_NUMBER(), RANK(), DENSE_RANK()
  • Value offset functions: LAG() and LEAD()

Theory

Like aggregates, window functions calculate aggregates across a partition of rows. However, unlike GROUP BY, they do not collapse rows into a single summary row. Every row in the input remains distinct in the result grid, alongside its corresponding partition value.

Key Window Syntax

  • OVER (PARTITION BY col) — Defines the slices of rows to compute the calculation within.
  • ROW_NUMBER() — Returns a unique index sequence (1, 2, 3...) per partition.
  • RANK() vs DENSE_RANK() — Handles values with matching sorting. Rank skips sequence slots (1, 2, 2, 4); Dense Rank does not (1, 2, 2, 3).
  • LAG() / LEAD() — Fetches values from the previous or next row in the partition (great for tracking period-over-period changes).
Window Functions window
-- 1. Rank employees by salary within their department
SELECT employee_id, first_name, department_id, salary,
       ROW_NUMBER() OVER (PARTITION BY department_id ORDER BY salary DESC) AS row_num,
       DENSE_RANK() OVER (PARTITION BY department_id ORDER BY salary DESC) AS sal_rank
FROM employees;

-- 2. Lag query: Compare this month's sales to last month's sales
SELECT sales_month, total_sales,
       LAG(total_sales, 1) OVER (ORDER BY sales_month) AS prev_month_sales
FROM monthly_sales;

Day 17 — Database Normalization

Today you will learn

  • What normalization is and why it prevents data anomalies
  • 1st Normal Form (1NF) — Atomic values
  • 2nd Normal Form (2NF) — Partial dependency removal
  • 3rd Normal Form (3NF) — Transitive dependency removal

Theory

Normalization is the systematic design process of organizing database columns and tables to eliminate data redundancy (which wastes disk space) and prevent database modification anomalies (insert, update, and delete issues).

The Normal Forms

FormRule RequirementAction to resolve
1NFAtomic values only. No repeating groups or multi-valued cells.Split comma-separated values into distinct rows.
2NFMust be in 1NF. Remove partial dependencies (columns depending on only part of a composite primary key).Move partial columns to their own lookup table.
3NFMust be in 2NF. Remove transitive dependencies (non-key columns depending on other non-key columns).If column B depends on column A, which depends on PK, move A and B to a separate table.

Day 18 — Transactions & Concurrency (ACID)

Today you will learn

  • What a database Transaction is
  • Understanding the ACID properties
  • Managing transactions with COMMIT and ROLLBACK
  • Transaction Isolation levels overview

Theory

A Transaction is a single logical unit of database work containing one or more SQL commands. (For example, transferring money: step 1 is deducting money from Account A, step 2 is adding money to Account B. If step 2 fails, step 1 must be reverted).

ACID Properties

  • Atomicity — "All or nothing". Either the entire transaction succeeds, or the database reverts every change.
  • Consistency — The database moves from one valid structural state to another, violating no schema constraints.
  • Isolation — Concurrent transactions run independently without interfering with each other's partial states.
  • Durability — Once committed, changes are written to non-volatile storage and survive power outages or crashes.
Transactions control tcl
-- 1. Start a database transaction (PostgreSQL syntax)
BEGIN;

-- 2. Perform updates
UPDATE accounts SET balance = balance - 100 WHERE account_id = 1;
UPDATE accounts SET balance = balance + 100 WHERE account_id = 2;

-- 3. Confirm modifications permanently
COMMIT;

-- Or if something failed mid-way:
-- ROLLBACK;

Day 19 — Stored Procedures & Triggers

Today you will learn

  • What Stored Procedures and Functions are
  • Writing a basic stored procedure (PL/pgSQL syntax)
  • Understanding Triggers (event-driven database operations)

Theory

Stored Procedures and Functions are SQL scripts compiled and stored directly inside the database catalog. They can contain control flow logic (loops, conditionals, variables) and are executed directly by the server, reducing network traffic.

Triggers

A Trigger is a special procedure that executes automatically in response to database modifications. Triggers can run: BEFORE or AFTER an INSERT, UPDATE, or DELETE command is processed on a table.

Procedure Example procs
-- 1. Create a function to add two numbers (PostgreSQL syntax)
CREATE FUNCTION fn_add_numbers(a INT, b INT) 
RETURNS INT AS $$
BEGIN
    RETURN a + b;
END;
$$ LANGUAGE plpgsql;

-- 2. Call the function
SELECT fn_add_numbers(10, 25);
